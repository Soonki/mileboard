## Layout Contract

### No New Visual Layout

This phase does not introduce new visual elements or layout changes. The board layout, lane layout, and card layout remain identical to Phase 7.

The only visual difference users will observe:

1. **When sort is "none" (default):** Cards within a lane can be dragged to reorder. The dnd-kit `SortableContext` + `verticalListSortingStrategy` (already present in Lane.tsx) provides the reorder animation automatically.
2. **When sort is active (field !== 'none'):** Cards within a lane cannot be dragged to reorder. The grab cursor is removed by dnd-kit's `disabled=true` on `useSortable`. Cards still respond to click (open in Backlog).

### Drop Placeholder Visual (dnd-kit built-in)

During intra-lane reorder, dnd-kit's `verticalListSortingStrategy` automatically animates adjacent cards to make space for the dragged item. This is the same mechanism already configured in Phase 5. No custom placeholder element is needed.

```
Lane (sort = none, reorder active)
+---------------------------+
| LaneHeader                |
+---------------------------+
| [Card A]                  |  <-- cards animate apart when
| [Card B] <- dragging      |      Card B is being dragged
|    ___gap___              |      to a new position
| [Card C]                  |
| [Card D]                  |
+---------------------------+

Lane (sort = active, reorder disabled)
+---------------------------+
| LaneHeader                |
+---------------------------+
| [Card A]  cursor: pointer |  <-- no grab handle, click only
| [Card B]  cursor: pointer |
| [Card C]  cursor: pointer |
| [Card D]  cursor: pointer |
+---------------------------+
```

## Interaction Contract

### Intra-Lane Reorder Flow (REORD-01)

1. User grabs a card within a lane (sort must be "none")
2. `PointerSensor` activates after 5px movement (existing `activationConstraint`)
3. `DragOverlay` shows the dragged card at 1.05x scale with `--shadow-drag` (existing Phase 5 behavior)
4. Adjacent cards animate to make space via dnd-kit's `verticalListSortingStrategy` transition
5. User drops the card at a new position within the same lane
6. `handleDragEnd` detects `fromLaneId === toLaneId` and calls `reorderStore.reorder(laneId, fromIndex, toIndex)`
7. Custom order is updated in memory and persisted to plugin-store
8. Card list re-renders in the new order

### Sort Mode Exclusivity (REORD-03, D-05, D-06)

1. When `sortStore.field !== 'none'`, all `useSortable` hooks receive `disabled: true` (in addition to existing `isMultiMilestone` check)
2. dnd-kit removes drag listeners when disabled, so the card's cursor falls back to `pointer` (click-to-open behavior remains)
3. Cross-lane DnD remains fully functional during sort mode (D-06). Only intra-lane reorder is disabled
4. No banner, tooltip, or explicit "sort mode" indicator is displayed (D-05). The cursor change provides sufficient feedback
5. When user switches sort back to "none" (`sortStore.field === 'none'`), the custom order is restored from reorderStore and intra-lane DnD re-enables

### Custom Order Persistence (REORD-02, D-03)

1. Custom order is stored as `Record<string, number[]>` mapping `laneId` to `issueId[]` array
2. On every reorder action, the full map is saved to plugin-store (fire-and-forget, same pattern as sortStorage)
3. On app startup, custom order is loaded from plugin-store before initial board render
4. If plugin-store data is corrupted or missing, custom order defaults to empty `{}` (falls back to keyId order per D-02)

### New Issue Merge (D-04)

1. When board data refreshes (API fetch), new issues not present in the custom order array are appended at the end of the lane's order
2. New issues are appended in keyId ascending order (consistent with D-02 default)
3. Issues removed from the API response (deleted/moved) are silently pruned from the custom order array
4. Existing manual order is preserved for issues that remain in the lane

### Cross-Lane Move + Order Update (D-07, D-08)

1. When a card is dragged from Lane A to Lane B (cross-lane move):
   - Lane A's custom order: the moved issue's ID is removed
   - Lane B's custom order: the moved issue's ID is appended at the end
2. Both lane orders are persisted to plugin-store in a single save
3. If Lane B has no custom order yet, a new order array is created containing existing issues (by keyId) plus the moved issue at the end

### Custom Order + Filter Interaction

1. When filters hide cards, the custom order array is NOT modified
2. Filtered-out cards retain their position in the custom order
3. When filters are cleared, cards reappear in their custom order positions
4. Reordering while filters are active: the user reorders only visible cards. Hidden cards retain their relative positions. The implementation merges visible-card order changes back into the full custom order array.

### Custom Order + Sort Interaction (D-01)

1. Activating a sort does NOT destroy the custom order in plugin-store
2. Sort mode overrides display order (sorted view takes precedence over custom order)
3. Deactivating sort (field = 'none') restores the custom order display
4. The custom order persists independently of sort state

## State Shape Contract

### ReorderStore (new Zustand store)

```typescript
type ReorderMap = Record<string, number[]>; // laneId -> issueId[]

interface ReorderStoreState {
  orderMap: ReorderMap;            // default: {}

  reorder: (laneId: string, fromIndex: number, toIndex: number) => void;
  setLaneOrder: (laneId: string, issueIds: number[]) => void;
  removeLaneOrder: (laneId: string) => void;
  loadFromStorage: () => Promise<void>;
}
```

- Default: `{ orderMap: {} }`
- Custom order IS persisted via plugin-store (REORD-02)
- Store key in plugin-store: `'reorder'` in `settings.json` file (alongside `'connection'` and `'sort'`)
- When `orderMap[laneId]` is undefined or empty, fallback to keyId ascending order (D-02)

### Sort Mode Detection

Read from existing `sortStore`:
- `sortStore.field === 'none'` --> custom order active, intra-lane DnD enabled
- `sortStore.field !== 'none'` --> sort active, intra-lane DnD disabled

No new state is added to sortStore.

### Keyboard Accessibility for Reorder

dnd-kit's `useSortable` provides built-in keyboard support:
- **Space/Enter** on a focused card: activates drag mode
- **Arrow Up/Down** while dragging: moves the card up/down in the list
- **Space/Enter** again: drops the card at the new position
- **Escape**: cancels the drag and returns the card to its original position

When sort is active (`disabled: true`), keyboard drag activation is suppressed. Tab navigation and click-to-open remain functional.
