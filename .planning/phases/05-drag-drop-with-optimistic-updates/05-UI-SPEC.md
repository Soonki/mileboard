## Interaction Contract: Drag & Drop

### Card States

| State | Visual Treatment | CSS |
|-------|-----------------|-----|
| Default | Current IssueCard style | Existing `.card` styles |
| Hover (no drag) | Background `#ECEEF0` | Existing `.card:hover` |
| Dragging (overlay) | Scale 1.05, `--shadow-drag`, fully opaque, border-radius preserved | New `.dragOverlay` class |
| Original position during drag | `display: none` (card removed from flow) | New `.cardDragging` class with `opacity: 0; pointer-events: none` |
| Multi-milestone (disabled) | `cursor: not-allowed` on drag attempt, card does not lift | New `.cardDragDisabled` class |

Source: CONTEXT.md D-01, D-02, D-08

### Lane States

| State | Visual Treatment | CSS |
|-------|-----------------|-----|
| Default | Current Lane style | Existing `.lane` styles |
| Drop target (active) | Background changes to `--color-drop-target` | New `.laneDropTarget` class |
| Drop target (invalid) | No visual change (multi-milestone cards cannot trigger drop on other lanes) | N/A |

Source: CONTEXT.md D-03

### DragOverlay Component Spec

The DragOverlay renders a clone of the IssueCard at the pointer position during drag.

| Property | Value |
|----------|-------|
| Scale | `transform: scale(1.05)` |
| Shadow | `--shadow-drag` |
| Opacity | 1.0 (fully opaque) |
| Border radius | `--radius-input` (6px, matches card) |
| Border | 1px solid `--color-border` |
| Background | `--color-surface` |
| Cursor | `grabbing` |
| Transition | `--transition-drag` for scale-in on drag start |

### Activation Constraint

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Type | Distance | Simpler than delay for desktop |
| Threshold | 5px | Distinguishes click from drag without noticeable lag |

## Interaction Contract: Error Toast (sonner)

### Toaster Provider Placement

Add `<Toaster />` from sonner at the Board level (inside Board.tsx or App.tsx).

| Property | Value |
|----------|-------|
| Position | `bottom-right` |
| Duration | 5000ms (5 seconds auto-dismiss) |
| Max visible | 3 (stack vertically) |
| Theme | `light` |
| Rich colors | `false` (use default sonner styling) |
| Close button | `true` (allow manual dismiss) |

Source: CONTEXT.md D-09, D-10, D-11

### Toast Content

| Property | Value |
|----------|-------|
| Type | `toast.error()` |
| Message language | Japanese (from Rust backend error strings) |
| Icon | Default sonner error icon (red circle with X) |

Source: CONTEXT.md D-11, D-12

### Toast Timing

Rollback animation completes before or simultaneously with toast display. The user sees the card snap back to its original lane AND the error message at the same time.

Source: CONTEXT.md D-12

## Interaction Contract: Optimistic Update Flow

### Visual Timeline

```
1. User drops card on target lane
   -> Card instantly appears in target lane (optimistic)
   -> Source lane removes card from its list
   -> Lane header counts update immediately

2a. API success
   -> No additional visual change (state already correct)

2b. API failure
   -> Card animates back to original lane (rollback)
   -> Source and target lane counts revert
   -> Error toast appears at bottom-right
```

### Rollback Animation

| Property | Value |
|----------|-------|
| Duration | `--transition-normal` (200ms) |
| Easing | `ease-out` |
| Behavior | Card re-appears in original position in original lane |

Source: CONTEXT.md D-16, D-17

## Copywriting Contract

| Element | Copy |
|---------|------|
| Empty lane (existing) | 課題なし |
| Empty lane during DnD hover | 課題なし (no change -- empty lane accepts drops) |
| Error toast: API failure on move | マイルストーンの変更に失敗しました: {error detail from backend} |
| Error toast: network error | ネットワークエラーが発生しました。接続を確認してください |
| Warning badge tooltip | 他のマイルストーン: {milestone names joined by ", "} |
| Multi-milestone drag attempt | No text -- cursor changes to `not-allowed` (D-08) |
| Unassigned lane name (existing) | 未割り当て |

No primary CTA button in this phase. The core interaction is direct manipulation (drag-and-drop), not button-driven.

No destructive actions requiring confirmation in this phase. Moving an issue between milestones is reversible (drag it back), and failed moves auto-rollback.
