# Milestone v1.0: MVP

**Status:** SHIPPED 2026-04-08
**Phases:** 1-5
**Total Plans:** 14

## Overview

Backlogのマイルストーンをカンバンレーンとして表示し、ドラッグ&ドロップで課題のマイルストーン移動を可能にするTauriデスクトップアプリのMVP。接続設定 → Rust APIパイプライン → カンバンボード → ボード強化 → DnDインタラクションの5フェーズで段階的に構築。

## Phases

### Phase 1: Foundation & Connection Settings

**Goal**: Users can configure and validate their Backlog connection, with settings persisted across app restarts
**Depends on**: Nothing (first phase)
**Requirements**: CONN-01, CONN-02, CONN-03
**Plans**: 3 plans

Plans:
- [x] 01-01: Backlog API service layer: type contracts, backlogApi.ts (testConnection + fetchProjects), Vitest infrastructure
- [x] 01-02: Project scaffold and persistence: Cargo.toml plugins, Tauri capabilities, settingsStore.ts (Zustand), settingsStorage.ts, global.css tokens
- [x] 01-03: Settings UI: SettingsForm (6 states), SettingsCard, SettingsModal, App.tsx routing, human verification

### Phase 2: Backlog Data Pipeline

**Goal**: The Rust backend reliably fetches milestones and issues from Backlog with rate-limit awareness and exposes them to the frontend via typed IPC commands
**Depends on**: Phase 1
**Requirements**: (infrastructure -- enables Phases 3-5)
**Plans**: 3 plans

Plans:
- [x] 02-01: Rust Backlog API client core: serde types, thiserror errors, BacklogClient with rate limiting and pagination
- [x] 02-02: Tauri IPC command wiring: fetch_board_data command, BacklogClient Managed State, lib.rs registration
- [x] 02-03: TypeScript type definitions and tauriBridge IPC proxy with invoke mock setup and unit tests

### Phase 3: Core Kanban Board

**Goal**: Users see their Backlog milestones as chronological lanes with issue cards, plus an unassigned lane, with loading feedback during data fetch
**Depends on**: Phase 2
**Requirements**: BOARD-01, BOARD-02, BOARD-03, UX-01
**Plans**: 3 plans

Plans:
- [x] 03-01: boardStore (Zustand state management) + Phase 3 CSS design tokens in global.css
- [x] 03-02: Leaf presentational components: StatusBadge, PriorityIndicator, IssueCard, EmptyLane with tests
- [x] 03-03: Board assembly: Lane, LaneHeader, BoardSkeleton, BoardError, BoardHeader, Board + App.tsx wiring + visual verification

### Phase 4: Board Enrichment

**Goal**: The board provides at-a-glance workload visibility through status colors, lane statistics, and direct linking to Backlog issues
**Depends on**: Phase 3
**Requirements**: BOARD-04, BOARD-05, BOARD-06, UX-03
**Plans**: 2 plans

Plans:
- [x] 04-01: Utilities (colorContrast, memberBreakdown) + StatusBadge color + MemberBreakdown component + IssueCard click/ripple
- [x] 04-02: LaneHeader enrichment (issue count, toggle, member breakdown) + Lane wiring + visual verification

### Phase 5: Drag & Drop with Optimistic Updates

**Goal**: Users can reassign issues between milestones by dragging cards across lanes, with instant visual feedback and safe rollback on failure
**Depends on**: Phase 4
**Requirements**: DND-01, DND-02, DND-03, UX-02
**Plans**: 3 plans

Plans:
- [x] 05-01: Rust update_issue_milestone + tauriBridge proxy + boardStore moveIssue with optimistic update/rollback
- [x] 05-02: @dnd-kit DnD integration: Board/Lane/IssueCard + WarningBadge + DragOverlayCard + sonner Toaster
- [x] 05-03: Regression tests + human visual/functional verification

## Milestone Summary

**Key Decisions:**
- Tauriデスクトップアプリとして構築 — CORS問題の根本解消 + 既存経験の活用
- Rust backendでBacklog API通信 — レート制限対応を含む
- @dnd-kit/core + sortable — モダン、アクセシブル、軽量DnDライブラリ
- 複数マイルストーン持ちはレーン間DnD禁止 — 異常状態の課題の意図しない操作を防止
- オプティミスティック更新 + snapshot/rollback — 即座のUI反映と安全なエラー回復
- Board側onDragOverでドロップターゲット管理 — useDroppableのisOver制限を回避

**Issues Resolved:**
- useDroppable isOverが同一レーン内ドラッグで発火しない問題 → Board側onDragOverで解決
- @dnd-kitパッケージのworktreeマージ後の未インストール → npm installで解決

**Issues Deferred:**
- settingsStorage.ts StoreOptions型エラー（Phase 1由来、機能影響なし）
- APIキーのセキュアストレージ（tauri-plugin-keyring互換性未確認）

**Technical Debt Incurred:**
- settingsStorage.ts の StoreOptions 型不一致（autoSave → defaults必須）
- REQUIREMENTS.md のトレーサビリティ表が一部Pending表記のまま
